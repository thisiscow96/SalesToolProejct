# 판매툴 문서 허브 (`docs/`)

이 페이지는 **저장소에 처음 들어온 사람(면접관·협업자)**이 *무엇을 어떻게 관리하는지* 바로 파악할 수 있도록 구성했습니다.

---

## 한눈에 보기

| 구분 | 내용 |
|------|------|
| **제품** | PC 웹(React) · 백엔드 API(Node) · 모바일(Expo) **모노레포** — 코드는 [`web/`](../web/) · [`server/`](../server/) · [`mobile/`](../mobile/) |
| **문서** | 아래 **00~99 번호 폴더**로 영역을 나누고, 설계·진척·테스트·배포·매뉴얼을 **같은 규칙**으로 쌓습니다. |
| **AI 보조 개발** | [Cursor](https://cursor.com)에서 **프로젝트 규칙**(`.cursor/rules/*.mdc`)을 두어, 코딩 에이전트가 **항상/특정 파일일 때** 따라야 할 관례를 자동으로 적용합니다. (규칙 목록은 [아래](#cursor-프로젝트-규칙-에이전트-하네싱)) |

---

## 이렇게 운영합니다

1. **Git / GitHub**  
   버전 관리·배포 이력·협업의 단일 진실 공급원입니다.

2. **번호 폴더 문서 (`docs/`)**  
   분석·설계 → 개발 준비 → 진척 → 테스트 → 매뉴얼 → 배포 순으로 **번호가 클수록 운영·검증에 가까운 문서**가 많습니다.  
   배포·객체 변경·진척은 **문서 규칙**에 맞춰 날짜·이력 형식을 맞춥니다.

3. **Cursor 프로젝트 규칙 (에이전트 하네싱)**  
   “에이전트에게 매번 설명하지 않고도” 같은 품질로 돌아가게 하려고, 저장소 루트의 **[`.cursor/rules/`](../.cursor/rules/)** 에 규칙 파일을 둡니다.  
   - **Always Apply**: 열린 파일과 무관하게 세션에 주입 (예: 모노레포 구조, UI 입력 가시성).  
   - **Glob**: 특정 경로 작업 시에만 (예: 배포일지 작성, 설계·진척 문서 편집, 마이그레이션 SQL).

아래 표는 **현재 두고 있는 규칙**과 역할입니다. (면접관: “규칙으로 무엇을 강제하는지”를 여기서 확인할 수 있습니다.)

### Cursor 프로젝트 규칙 (에이전트 하네싱)

| 규칙 파일 | 적용 | 하는 일 (요약) |
|-----------|------|----------------|
| [`monorepo-structure.mdc`](../.cursor/rules/monorepo-structure.mdc) | 항상 | `web` / `server` / `mobile` 역할·공통 관례 |
| [`ui-ux-input-visibility.mdc`](../.cursor/rules/ui-ux-input-visibility.mdc) | 항상 | 테이블·입력칸에서 값이 잘리지 않게 UI 점검 |
| [`render-database-migrate.mdc`](../.cursor/rules/render-database-migrate.mdc) | 항상 + `server/migrations/**` | 운영 DB 마이그 시 비밀 미커밋·Actions 절차 |
| [`deployment-log.mdc`](../.cursor/rules/deployment-log.mdc) | 배포일지 `*.md` | 배포 시 DEP ID·반영 시스템 등 일지 형식 |
| [`design-and-progress-docs.mdc`](../.cursor/rules/design-and-progress-docs.mdc) | 설계·진척 관련 문서 | 객체 변경 사유·진척일지·설계 요약 관례 |

> **용어 정리**: “에이전트 하네싱”은 **에이전트(코딩 AI)가 따라야 할 내부 규칙을 코드베이스에 두고**, 채팅마다 반복 설명하지 않고 **일관된 출력**을 내게 하는 방식을 가리킵니다. (Skill·User rules는 로컬/계정 설정이고, **이 레포에 커밋되는 것은 주로 `.cursor/rules/`** 입니다.)
