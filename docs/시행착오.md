# 배포·연동 시행착오 정리

프로젝트 배포 및 이메일 인증 연동 과정에서 겪은 문제와 해결을 정리한 문서입니다.

---

## 1. 이메일 인증 — Gmail SMTP가 PaaS에서 안 됨

### 현상
- 회원가입 시 "인증번호 발송" 클릭 시 **발송 중**에서 멈추거나, **테스트용 인증번호**만 표시되고 실제 메일이 오지 않음.
- 로그/응답: **ETIMEDOUT** (Connection timeout).

### 원인
- **Railway**, **Render** 모두 아웃바운드 **SMTP(587, 465)** 를 막거나 제한함.
- Gmail SMTP(smtp.gmail.com)로 나가는 TCP 연결이 PaaS 방화벽에서 차단되어 타임아웃 발생.
- "Render로 옮기면 된다"고 안내했으나, **Render도 SMTP를 막는지 사전 검증하지 않음** → 동일 증상 재발.

### 해결
- **Resend** 등 **HTTPS 기반 이메일 API** 사용. SMTP 포트 없이 API 호출만으로 발송 가능.
- `RESEND_API_KEY` 설정 시 Resend로 발송, 미설정 시 로컬에서는 기존 Gmail SMTP(nodemailer) 사용.

### 교훈
- 호스팅을 "이메일 되게 하려고" 바꿀 때는, **해당 플랫폼의 아웃바운드 SMTP 허용 여부**를 먼저 확인한 뒤 권장할 것.

## 2. 이메일 중복 확인 — 처음 입력한 이메일도 "이미 사용 중"

### 현상
- DB에 해당 이메일이 없는데도 "이미 사용 중인 이메일입니다" 표시.

### 원인
- **프론트**: 서버 오류(500, 네트워크 오류 등) 시에도 `data.available === false`이면 무조건 "이미 사용 중"으로 표시하고 있음.
- **백엔드**: 500/에러 응답에 사용자용 메시지가 없어, 클라이언트가 원인 구분 불가.

### 해결
- `!res.ok` 일 때는 "확인할 수 없습니다" 등으로 표시하고, "이미 사용 중"은 **200 응답이면서 `available === false`일 때만** 표시하도록 수정.
- 백엔드 500 응답에 `message`, 필요 시 `error`(디버깅용) 필드 포함.

### 교훈
- 에러 응답과 정상 응답을 구분해서 처리하고, 실패 시 사용자 메시지를 명확히 할 것.

## 3. Vercel 배포 — API 요청 405 / 500

### 현상
- 배포된 사이트에서 API 호출 시 **405 Method Not Allowed** 또는 **500 Internal Server Error**.
- 요청 URL이 `https://sales-tool-proejct.vercel.app/api/...` 로 나감 (프론트와 같은 도메인).

### 원인
- **405**: `vercel.json` 의 rewrite가 없거나, `/api` 를 백엔드가 아닌 정적 호스트로 보냄. Vercel에는 API 서버가 없어 405 발생.
- **500**: rewrite는 있으나 **destination** 이 **Railway** 등 예전 백엔드 URL로 되어 있어, 해당 서버 오류 또는 DB 없음(relation "user" does not exist) 등으로 500 발생.
- 백엔드를 **Render**로 바꾼 뒤에도 **vercel.json** 을 Render URL로 갱신하지 않아, 계속 옛 백엔드로 요청이 감.

### 해결
- `vercel.json` 에서 `/api/:path*` 의 **destination** 을 실제 사용 중인 백엔드 URL로 설정 (예: `https://salestoolproejct.onrender.com/api/:path*`).
- 백엔드 호스팅을 바꿀 때마다 **Vercel 환경 변수 `VITE_API_URL`** 및 **vercel.json rewrite** 를 함께 갱신.

### 교훈
- "백엔드만 바꾼다"고 해도, 프론트/프록시가 **어디로 요청을 보내는지** 한 번 더 확인할 것.

## 4. Render — DB 연결 실패 (마이그레이션 실패)

### 현상
- Web Service 배포 시 `[migrate] DB 연결 실패:` 후 **Exited with status 1**.

### 원인
- **External Database URL** 사용. Render 내부 통신에는 **Internal Database URL** 이 필요함.
- 또는 Postgres 서비스를 Web Service에 **연결(Link)** 하지 않아 `DATABASE_URL` 가 주입되지 않음.
- Render Postgres는 SSL 필요 시 `sslmode=require` 등이 붙은 URL인데, 마이그레이션 스크립트에서 SSL 옵션 미처리.

### 해결
- Postgres **Info** 탭에서 **Internal Database URL** 복사 후 Web Service **Environment** 에 `DATABASE_URL` 로 설정.
- 또는 Web Service **Environment** 에서 **Connect existing resource** 로 Postgres 연결해 `DATABASE_URL` 자동 주입.
- 마이그레이션 스크립트에서 `render.com` 호스트 또는 `sslmode` 존재 시 `ssl: { rejectUnauthorized: false }` 적용.

### 교훈
- PaaS에서 DB 연결 시 **Internal vs External URL** 구분과 **리소스 연결** 절차를 문서에 명시할 것.

## 5. 데이터 조회 — user id vs 중매인 번호(agent_no)

### 현상
- 요구사항: **중매인 번호(agent_no)** 기준으로 데이터 조회. 기존에는 **user id** 로만 조회하고 있음.

### 해결
- 백엔드: 헤더 **X-Agent-No**(중매인 번호)를 받아, `"user"` 테이블에서 `agent_no` 로 **user id** 를 조회한 뒤 기존처럼 user_id 로만 쿼리.
- 프론트: `authHeaders()` 에서 `X-User-Id` 대신 **X-Agent-No: user.agent_no** 전송.
- DB 스키마는 그대로 user_id 기반 유지.

### 교훈
- "식별자"를 바꿀 때는 API 계약(헤더/파라미터)과 DB 조회 로직을 함께 바꾸고, 클라이언트가 보내는 값과 서버가 기대하는 값을 맞출 것.
