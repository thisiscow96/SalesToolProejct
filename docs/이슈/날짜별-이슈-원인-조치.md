# 이슈 — 날짜별 정리 (이슈 원인 · 조치 방법)

프로젝트에서 발생한 이슈를 **발생 시기(단계)** 별로 정리하고, **이슈 원인**과 **조치 방법**만 요약한 문서입니다.  
상세 현상·교훈은 [배포·연동 시행착오 정리](../시행착오.md)를 참고하세요.

---

## 배포·연동 단계

### 1. 이메일 인증 — Gmail SMTP가 PaaS에서 안 됨 (ETIMEDOUT)

| 항목 | 내용 |
|------|------|
| **이슈 원인** | Railway·Render 모두 아웃바운드 SMTP(587, 465)를 막고 있어, Gmail SMTP(smtp.gmail.com) 연결이 PaaS 방화벽에서 차단되어 ETIMEDOUT 발생. Render로 옮긴 뒤에도 SMTP 미허용으로 동일 증상. |
| **조치 방법** | Resend 등 HTTPS 기반 이메일 API 사용. `RESEND_API_KEY` 설정 시 Resend로 발송, 미설정 시 로컬에서만 Gmail SMTP(nodemailer) 사용. |

---

### 2. 이메일 중복 확인 — 처음 입력한 이메일도 "이미 사용 중"

| 항목 | 내용 |
|------|------|
| **이슈 원인** | 프론트에서 서버 오류(500, 네트워크 오류) 시에도 `data.available === false`이면 무조건 "이미 사용 중"으로 표시. 백엔드 500 응답에 사용자용 메시지 없어 원인 구분 불가. |
| **조치 방법** | `!res.ok` 일 때는 "확인할 수 없습니다" 등으로 표시하고, "이미 사용 중"은 **200 응답이면서 `available === false`일 때만** 표시. 백엔드 500 응답에 `message`(필요 시 `error`) 필드 포함. |

---

### 3. Vercel 배포 — API 요청 405 / 500

| 항목 | 내용 |
|------|------|
| **이슈 원인** | 405: `vercel.json` rewrite 없거나 `/api`가 백엔드가 아닌 정적 호스트로 감. 500: rewrite destination이 예전 백엔드(Railway 등) URL로 되어 있거나, Render로 바꾼 뒤에도 `vercel.json`·`VITE_API_URL` 미갱신으로 옛 백엔드로 요청 감. |
| **조치 방법** | `vercel.json`에서 `/api/:path*` destination을 실제 백엔드 URL로 설정(예: `https://salestoolproejct.onrender.com/api/:path*`). 백엔드 호스팅 변경 시 **Vercel 환경 변수 `VITE_API_URL`** 와 **vercel.json rewrite** 를 함께 갱신 후 Redeploy. |

---

### 4. Render — DB 연결 실패 (마이그레이션 실패)

| 항목 | 내용 |
|------|------|
| **이슈 원인** | External Database URL 사용 또는 Postgres를 Web Service에 연결하지 않아 `DATABASE_URL` 미주입. Render Postgres SSL URL인데 마이그레이션 스크립트에서 SSL 옵션 미처리. |
| **조치 방법** | Postgres **Info** → **Internal Database URL** 복사 후 Web Service **Environment**에 `DATABASE_URL` 설정. 또는 **Connect existing resource**로 Postgres 연결해 자동 주입. 마이그레이션 스크립트에서 `render.com` 호스트 또는 `sslmode` 존재 시 `ssl: { rejectUnauthorized: false }` 적용. |

## 기능·요구사항 반영 단계

### 5. 데이터 조회 — user id vs 중매인 번호(agent_no)

| 항목 | 내용 |
|------|------|
| **이슈 원인** | 요구사항은 중매인 번호(agent_no) 기준 조회인데, 기존에는 user id로만 조회하고 있음. |
| **조치 방법** | 백엔드: 헤더 **X-Agent-No**로 agent_no 수신 후 `"user"` 테이블에서 agent_no로 user id 조회해 기존 user_id 기반 쿼리 유지. 프론트: `authHeaders()`에서 **X-Agent-No: user.agent_no** 전송. DB 스키마는 user_id 기반 유지. |

## 문서·운영 정리

### 6. 문서·배포 정리

| 항목 | 내용 |
|------|------|
| **이슈 원인** | 배포·환경 변수·시행착오가 문서에 흩어져 있어 설정 누락·재발 방지가 어려움. |
