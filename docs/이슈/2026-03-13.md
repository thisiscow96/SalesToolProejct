# 이슈 — 2026-03-13

## 1. Render — DB 연결 실패 (마이그레이션 실패)

### 날짜
2026-03-13

### 무슨 이슈였는지 (현상)
- Web Service 배포 시 `[migrate] DB 연결 실패:` 후 **Exited with status 1**.

### 이슈 원인
- **External Database URL** 사용. Render 내부 통신에는 **Internal Database URL** 이 필요함.
- 또는 Postgres 서비스를 Web Service에 **연결(Link)** 하지 않아 `DATABASE_URL` 가 주입되지 않음.
- Render Postgres는 SSL 필요 시 `sslmode=require` 등이 붙은 URL인데, 마이그레이션 스크립트에서 SSL 옵션 미처리.

### 조치 방법
- Postgres **Info** 탭에서 **Internal Database URL** 복사 후 Web Service **Environment** 에 `DATABASE_URL` 로 설정.
- 또는 Web Service **Environment** 에서 **Connect existing resource** 로 Postgres 연결해 `DATABASE_URL` 자동 주입.
- 마이그레이션 스크립트에서 `render.com` 호스트 또는 `sslmode` 존재 시 `ssl: { rejectUnauthorized: false }` 적용.

### 교훈
- PaaS에서 DB 연결 시 **Internal vs External URL** 구분과 **리소스 연결** 절차를 문서에 명시할 것.

## 2. 데이터 조회 — user id vs 중매인 번호(agent_no)

### 날짜
2026-03-13

### 무슨 이슈였는지 (현상)
- 요구사항: **중매인 번호(agent_no)** 기준으로 데이터 조회. 기존에는 **user id** 로만 조회하고 있음.

### 이슈 원인
- API·DB가 user_id 기반으로만 구성되어 있어, agent_no 기준 조회로 바꿀 필요 있음.

### 조치 방법
- **백엔드**: 헤더 **X-Agent-No**(중매인 번호)를 받아, `"user"` 테이블에서 `agent_no` 로 **user id** 를 조회한 뒤 기존처럼 user_id 로만 쿼리.
- **프론트**: `authHeaders()` 에서 `X-User-Id` 대신 **X-Agent-No: user.agent_no** 전송.
- DB 스키마는 그대로 user_id 기반 유지.

### 교훈
- "식별자"를 바꿀 때는 API 계약(헤더/파라미터)과 DB 조회 로직을 함께 바꾸고, 클라이언트가 보내는 값과 서버가 기대하는 값을 맞출 것.
