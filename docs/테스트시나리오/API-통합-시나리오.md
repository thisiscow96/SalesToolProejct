# API 통합 테스트 시나리오

백엔드(`server/index.js`) 기준으로, **샘플 API 호출**로 정상 동작을 검증할 때 따를 수 있는 시나리오입니다.  
(실제 자동 테스트 코드는 이 문서의 **시나리오 ID**를 그대로 테스트 이름에 쓰는 것을 권장합니다.)

---

## 0. 공통 전제

| 항목 | 내용 |
|------|------|
| DB | `schema-full.sql` 적용 또는 `001`~`004` 마이그레이션 완료 |
| 서버 | `npm start` 등으로 API 기동 (예: `http://localhost:3000`) |
| 인증 | 보호 API는 헤더 `X-Agent-No: {활성 사용자의 agent_no}` 필수 |
| Content-Type | `application/json` (POST 본문 사용 시) |

**시드/준비 데이터(권장)**  
시나리오를 한 줄로 끝까지 돌리려면 다음이 있으면 좋습니다.

- `user`: `status=active`, 알려진 `agent_no`
- `account` 2건 이상: `user_id` 동일, `type=supplier`(또는 `wholesaler` 등 매입 허용 타입) 1건, `type=customer`(또는 매출 허용 타입) 1건
- `product`: 최소 1건

> 계정·거래처가 없으면 **로그인·거래처 조회** 시나리오는 통과하고, **매입/매출** 시나리오는 스킵하거나 실패로 기록합니다.

## 2. 시나리오 상세

### S-001 — 헬스 체크

| 항목 | 내용 |
|------|------|
| **목적** | 서버·DB 연결 가능 여부 |
| **메서드·경로** | `GET /health` |
| **헤더** | 없음 |
| **기대** | `200`, `ok: true`, `db: "connected"` (DB 설정된 환경) |

---

### S-002 — 보호 API 미인증

| 항목 | 내용 |
|------|------|
| **목적** | 로그인 헤더 없을 때 거절 |
| **메서드·경로** | `GET /api/partners` (또는 `/api/inventory`) |
| **헤더** | `X-Agent-No` 없음 |
| **기대** | `401`, 본문에 로그인 필요 메시지 |

---

### S-003 — 잘못된 중매인 번호

| 항목 | 내용 |
|------|------|
| **목적** | 존재하지 않는 `agent_no` |
| **메서드·경로** | `GET /api/partners` |
| **헤더** | `X-Agent-No: NO_SUCH_AGENT_99999` |
| **기대** | `401` |

---

### S-004 — 거래처 목록

| 항목 | 내용 |
|------|------|
| **목적** | 회원 소유 거래처만 조회 |
| **메서드·경로** | `GET /api/partners` |
| **헤더** | 유효한 `X-Agent-No` |
| **기대** | `200`, `data` 배열, 항목에 `id`, `name`, `type` 등 |

---

### S-005 — 상품 목록

| 항목 | 내용 |
|------|------|
| **목적** | 상품 마스터 조회 |
| **메서드·경로** | `GET /api/products` |
| **헤더** | 없어도 됨(현재 구현 기준) |
| **기대** | `200`, `data` 배열 |

---

### S-006 — 재고 목록

| 항목 | 내용 |
|------|------|
| **목적** | 로그인 사용자 기준 재고 |
| **메서드·경로** | `GET /api/inventory` |
| **헤더** | 유효한 `X-Agent-No` |
| **기대** | `200`, `data` 배열 (`quantity > 0` 만 보이는 현재 정책이면 빈 배열 가능) |

---

### S-007 — 매입 등록 (정상)

| 항목 | 내용 |
|------|------|
| **목적** | 입고 + 재고 증가 + `product_transfer` |
| **메서드·경로** | `POST /api/purchases` |
| **헤더** | `X-Agent-No`, `Content-Type: application/json` |
| **본문 예** | `{ "partner_id": <매입가능거래처ID>, "product_id": <상품ID>, "quantity": 10, "unit_price": 1000, "purchase_date": "2026-03-13", "memo": "시나리오" }` |
| **기대** | `201`, `data.id` 존재, 이후 `GET /api/inventory`에서 해당 상품 수량 증가(또는 신규 행) |

---

### S-008 — 매입 등록 (거래처 타입 오류)

| 항목 | 내용 |
|------|------|
| **목적** | 판매처만 있을 때 매입 차단 |
| **본문** | S-007과 동일하되 `partner_id`를 `type=customer` 인 계정으로 |
| **기대** | `400`, 매입 불가 메시지 |

---

### S-009 — 매입 등록 (유효성)

| 항목 | 내용 |
|------|------|
| **목적** | 필수 필드 검증 |
| **본문** | `quantity` 누락 또는 0, 또는 `purchase_date` 누락 |
| **기대** | `400` |

---

### S-010 — 매입 목록 (배분 필드)

| 항목 | 내용 |
|------|------|
| **목적** | `remaining_qty`, `allocated_qty` 노출 |
| **메서드·경로** | `GET /api/purchases?from_date=...&to_date=...` |
| **기대** | 각 행에 `remaining_qty`, `allocated_qty` (마이그레이션 미적용 시 쿼리 오류 가능 — 전제 점검) |

---

### S-011 — 매입 → 매출 전환 (정상)

| 항목 | 내용 |
|------|------|
| **목적** | `purchase_allocation` + 매출 + 재고 감소 |
| **전제** | S-007로 만든 매입에 `remaining_qty` ≥ 전환 수량, 재고 ≥ 전환 수량 |
| **메서드·경로** | `POST /api/purchases/convert-to-sales` |
| **본문 예** | `{ "items": [ { "purchase_id": <ID>, "partner_id": <판매가능거래처ID>, "quantity": 3, "unit_price": 1500, "sale_date": "2026-03-13" } ] }` |
| **기대** | `201`, `data`에 `sale_id`, `GET /api/sales`에 신규 행, 재고 감소, `GET /api/purchases`에서 해당 매입 `remaining_qty` 감소 |

---

### S-012 — 매출 전환 (잔여 초과)

| 항목 | 내용 |
|------|------|
| **목적** | 매입 잔여보다 많이 팔 수 없음 |
| **본문** | `remaining_qty`보다 큰 `quantity` |
| **기대** | `4xx/5xx`, 남은 수량 관련 메시지, 트랜잭션 롤백으로 데이터 불변 |

| 항목 | 내용 |

| **기대** | `400` 또는 `500`, 재고 부족 메시지 |

---

### S-014 — 매출 목록

| 항목 | 내용 |
|------|------|
| **목적** | 환불·상태 필드 |
| **메서드·경로** | `GET /api/sales?from_date=...&to_date=...` |
| **기대** | `status`, `refunded_qty` 등 (스키마 반영 시) |

---

### S-015 — 수금 등록 (정상)

| 항목 | 내용 |
|------|------|
| **목적** | `payment` + `payment_allocation` + sale `paid_amount` |
| **전제** | S-011로 만든 매출, 거래처 동일, 아직 미수 상태 |
| **메서드·경로** | `POST /api/payments` |
| **본문 예** | `{ "partner_id": <매출과동일거래처>, "amount": 4500, "paid_at": "2026-03-13", "entry_kind": "receive", "allocations": [ { "sale_id": <ID>, "amount": 4500 } ] }` |
| **기대** | `201`, `GET /api/sales` 해당 건 `paid_amount` 증가, `payment_status` 갱신 |

| **기대** | `500` 또는 에러 메시지(현재 구현은 throw) |

### S-018 — 환불 (정상)

## 3. 권장 실행 순서 (샘플 한 번에 검증)

한 사용자·한 상품·두 거래처(매입/매출)를 기준으로 **순서대로** 호출하면 끊김이 적습니다.

1. S-001  
2. S-002 (선택)  
3. S-004 ~ S-006 (유효 `X-Agent-No`로)  
4. S-007  
5. S-010 (매입 잔여 확인)  
6. S-011  
7. S-014 (매출 확인)  
8. S-015  
9. S-018 (부분 환불)  
10. S-020 (남은 재고 일부 폐기)  
11. S-008, S-009, S-012, S-016, S-019, S-021 (실패 케이스 샘플)

## 4. 자동화 시 매핑 제안

| 도구 | 제안 |
|------|------|
| 파일 위치 | 예: `server/test/api.integration.test.js` 또는 `server/__tests__/api-scenarios.test.js` |
| 이름 | `describe('S-007 매입 등록')` … `it('정상')` |
| 데이터 | `.env.test` + 테스트 전용 DB 또는 트랜잭션 롤백 |

이 문서만 먼저 유지보수하고, 테스트 코드는 시나리오 ID 주석으로 연결해 두면 **요구사항 ↔ 검증** 추적이 쉬워집니다.
