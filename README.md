# Sales Tool Project

> **매입·매출·재고·수금** 흐름을 한 화면에서 이어 붙여, **반복 입력·데이터 분산**으로 생기던 누락과 불일치를 줄이기 위한 **실무형 판매 관리 시스템**입니다.

### 핵심 개선 (한눈에)

| | |
|--|--|
| **업무** | 반복 입력 감소, 단계 간 데이터 재입력 최소화 |
| **데이터** | 매입/매출/재고/수금을 **같은 객체·이력**으로 추적 가능한 구조 |
| **운영** | 배포·설계·진척·테스트 문서를 **번호 규칙**으로 쌓아 유지보수·온보딩 비용 절감 |

### 역할

**분석 → 설계 → 구현 → 배포·운영 문서화**까지 End-to-End로 수행했습니다.

## Problem

- 매입·매출·재고·수금 정보가 **도구·시트·기억**에 흩어지면 입력이 반복되고, **누락·불일치**가 나기 쉽습니다.
- 나중에 **왜 이렇게 됐는지** 추적하려면 설계·배포·변경 이력이 남아 있어야 하는데, 팀에 맞는 **문서 구조**가 없으면 유지보수가 어렵습니다.

---

## Solution — Before vs After

| Before | After |
|--------|--------|
| 매입/매출/수금·재고가 **분산**되어 같은 값을 여러 번 입력 | **단일 흐름**으로 등록·조회·집계 (탭·모달 기준 화면 설계) |
| 반복 입력·수기 정리로 **피로도·오류** 증가 | 입력 단계·컬럼 가시성(긴 값도 안 잘리게) 등 **UI 규칙**을 두고 개선 |
| 변경·배포 **이력 추적 어려움** | 배포일지(DEP)·진척일지·시나리오·object 설계서로 **추적 가능한 문서 구조** |

## Implementation (요약)

| 영역 | 내용 |
|------|------|
| **구조** | 모노레포 — [`web/`](web/) (React·Vite), [`server/`](server/) (Express·PostgreSQL), [`mobile/`](mobile/) (Expo) |
| **품질·운영** | API 시나리오·배포 자동화 연계, DB 마이그레이션·비밀 관리 절차 정리 |
| **AI 보조 개발** | [`.cursor/rules/`](.cursor/rules/)에 프로젝트 규칙을 두어 에이전트 출력을 일관되게 유지 (상세는 [docs/README.md](docs/README.md)) |

---

## Impact

- **실사용·데모 기준**으로 입력·조회 동선을 짧게 가져가 **편의성**을 우선했습니다.
- 반복 작업·불필요한 재입력을 줄여 **업무 피로**를 낮추는 방향으로 화면·API를 맞췄습니다.
- 설계·배포·테스트를 **문서와 ID(DEP 등)**로 남겨, 이후 기능 추가·장애 대응 시 **근거를 찾기 쉬운** 상태를 목표로 했습니다.
